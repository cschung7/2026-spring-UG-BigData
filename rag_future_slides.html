RAG와의 관계
대체? 공존?
  • 단순 문서 검색 → Titans가 대체 가능
  • 동적 외부 DB 업데이트 → RAG 여전히 필요
  • 추론 집약 태스크 → Titans가 확실히 강함
  • 현실적: 하이브리드 구조
경제학 analogie: Titans의 Long-term Memory는 중앙은행이 과거 정책 결정 데이터를 모두 외우고 있는 것. RAG는 그때그때 보고서를 꺼내 읽는 것. 둘 다 필요한 상황이 있다.

문제: KV Cache란?
왜 이게 병목인가?

LLM이 토큰을 생성할 때마다 이전 모든 Key·Value 벡터를 저장. 70B 모델 512명 동시 사용 시 KV cache만 512GB — 모델 가중치의 4배.

TurboQuant 작동 원리
2단계 압축 파이프라인
  • PolarQuant — 벡터를 극좌표로 변환, 대부분의 압축 처리
  • QJL — 1비트 잔차 오류 보정으로 attention 정확도 유지
RAG와의 관계
RAG 파이프라인을 직접 강화
  • 긴 context → 더 많은 chunk 동시 처리 가능
  • 벡터 서치 속도 향상 (RAG 검색 단계)
  • 더 많은 사용자 동시 서빙 가능
  • RAG를 대체하지 않고 인프라 개선

QE ≡ KE 증명

쿼리 확장 = 키 확장 (수학적 동치). 따라서 query-time 이득을 index에 amortize 가능. 수렴도 이론적으로 보장.

Zipf 분포 활용
왜 효율적인가?

실세계 쿼리는 Zipf 분포 → 소수 문서에 트래픽 집중. 모든 key를 업데이트하지 않고 lazy update로 충분.

RAG와의 관계
RAG 위에 얹는 레이어

기존 RAG 파이프라인 수정 없이 추가. 쓸수록 좋아지는 자기진화 RAG. Training-free.

경제학 analogie (베이지안 업데이트): 사전 확률(기존 index) + 검증된 데이터(정답 쿼리) → 사후 확률(진화된 index). 단, norm-bounded이므로 과잉반응 없는 합리적 기대 조정.

결론 1 — RAG는 죽지 않았다: 세 연구 모두 RAG를 대체하지 않는다. Titans는 아키텍처 혁신, TurboQuant는 인프라 효율화, ERM은 인덱스 진화 — 모두 RAG 생태계를 강화하는 방향.
결론 2 — 레이어별 혁신: HW(TurboQuant) · 모델(Titans) · 소프트웨어(ERM)가 동시에 진화 중. 실무에서는 목적에 따라 레이어를 선택적으로 조합하는 전략이 핵심.
07  APPLICATION

금융·경제 분야 실제 적용 시나리오

시나리오 A — ERM 적용
리서치 리포트 DB

증권사·연구기관이 반복적으로 조회하는 섹터/기업 쿼리가 쌓일수록 관련 인덱스 키 자동 강화. 애널리스트 쿼리 패턴이 시스템을 훈련시킴.

시나리오 B — TurboQuant 적용
멀티유저 금융 AI 서비스

동일 GPU로 처리 가능한 동시 사용자 수 대폭 증가. 긴 공시문서·계약서를 한번에 처리하는 long-context RAG 서비스 가능.

시나리오 C — Titans 적용
복잡한 추론 질의

"2008 금융위기 패턴과 현재 상황의 유사점을 수백 페이지 보고서에서 찾아줘" 같은 멀티홉 장기 추론 태스크.

# 실무 판단 가이드 — 어떤 혁신을 언제 쓸까?

if 문제 == "긴 문서 추론 / 2M 토큰 처리":
    → Titans 기반 모델 검토 # 아직 상용화 초기

elif 문제 == "GPU 비용 / 동시 사용자 확장":
    → TurboQuant KV cache 압축 적용 # 커뮤니티 구현 활용

elif 문제 == "반복 쿼리 정확도 / 인덱스 개선":
    → ERM 레이어 추가 # 기존 RAG 스택 유지

else:
    → Standard RAG # 여전히 강력한 baseline
08  DISCUSSION & SUMMARY

토론 질문과 핵심 요약

토론 1
TurboQuant의 역설

KV cache 압축이 메모리 비용을 줄이면, 오히려 더 긴 context를 쓰게 되어 총 메모리 수요가 늘 수도 있다. (Jevons Paradox) 이게 왜 문제이거나 문제가 아닌가?

토론 2
Titans가 완벽하다면?

모델이 모든 것을 기억할 수 있다면 RAG가 필요한가? 어떤 use case에서는 여전히 RAG가 필수적인가? (힌트: 실시간성, 프라이버시, 비용)

3줄 요약
이번 강의 핵심
  • Titans — 모델이 직접 장기 기억, RAG 부분 대체 가능
  • TurboQuant — HW 병목 해소, RAG 인프라 비용 절감
  • ERM — 인덱스 자기진화, 기존 RAG를 점진적 강화
Big Picture: RAG는 "죽은" 것이 아니라 레이어별로 진화하고 있다. HW·아키텍처·소프트웨어가 동시에 발전하며 서로를 강화한다. 어떤 단일 기술도 모든 문제를 해결하지 않는다.
